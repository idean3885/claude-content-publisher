# 마크다운 가독성 규칙

기술 블로그 및 위키 포스팅의 마크다운 가독성을 일관되게 유지하기 위한 규칙 체계.

> **근거**: Google Developer Documentation Style Guide, Microsoft Writing Style Guide,
> Nielsen Norman Group 웹 가독성 연구, WCAG CJK 타이포그래피 권장사항 종합.

---

## 1. 문단 구조

### P1. 1문단 = 1아이디어

문단은 하나의 아이디어만 다룬다. 아이디어가 바뀌면 문단을 나눈다.

> **근거**: Google Developer Docs — "Each paragraph should focus on one idea."
> 독자는 문단 단위로 스캔하며, 복합 아이디어 문단은 핵심을 놓치게 한다.

Bad:
```markdown
Virtual Thread는 경량 스레드입니다. 기존 플랫폼 스레드와 달리 OS 스레드에 1:1로
매핑되지 않습니다. Spring Boot 3.2부터 설정 한 줄로 활성화할 수 있습니다.
`spring.threads.virtual.enabled=true`를 application.yml에 추가하면 됩니다.
```

Good:
```markdown
Virtual Thread는 경량 스레드입니다. 기존 플랫폼 스레드와 달리 OS 스레드에 1:1로
매핑되지 않습니다.

Spring Boot 3.2부터 설정 한 줄로 활성화할 수 있습니다.
`spring.threads.virtual.enabled=true`를 application.yml에 추가하면 됩니다.
```

### P2. 문단 길이 2-4문장

1문장은 파편화, 6문장 초과는 과밀.

> **근거**: NNG — 웹 사용자는 전체 텍스트의 20-28%만 읽는다.
> 짧은 문단이 스캔 효율을 높인다.

판단 기준:
- 1문장 문단 → P4(단문 문단) 규칙에 해당하는가? 아니면 다음 문단과 합칠 수 있는가?
- 5문장 이상 → 아이디어를 분리할 수 있는가? 뒤쪽 문장이 새로운 관점인가?

### P3. 주제문 선행

문단의 첫 문장이 핵심이다. 독자는 첫 문장 이후 건너뛸 수 있다.

> **근거**: NNG — "Users scan in an F-shaped pattern."
> 첫 문장이 핵심이 아니면 독자는 문단 전체를 건너뛴다.

Bad:
```markdown
여러 방법을 시도해 보았습니다. Lettuce 커넥션 풀을 조정하고, 타임아웃을 변경하고,
캐시 TTL도 수정했습니다. 결국 문제의 원인은 DNS 해석 지연이었습니다.
```

Good:
```markdown
문제의 원인은 DNS 해석 지연이었습니다. Lettuce 커넥션 풀, 타임아웃, 캐시 TTL을
순서대로 의심했지만, 최종적으로 DNS resolve 시간이 병목이었습니다.
```

### P4. 단문 문단은 강조/전환/결론에만

1-2문장 문단이 본문 흐름 중간에 반복되면 파편화 인상을 준다.

허용되는 경우:
- 섹션 도입부 (전환)
- 핵심 결론 강조
- 인용/경고/참고 블록

### P5. 열거 vs 산문

3개 이상 나열은 목록으로, 인과/맥락 설명은 산문으로.

> **근거**: Microsoft Writing Style Guide — "Use a list for three or more items."

판단 기준:
- 항목 간 순서/인과 관계가 없다 → 비순서 목록 (`-`)
- 항목 간 순서가 있다 → 순서 목록 (`1.`)
- 항목에 설명이 필요하다 → 산문 또는 정의 목록

Bad:
```markdown
설정해야 할 것은 데이터소스 URL, 커넥션 풀 크기, 타임아웃, 유휴 연결 수입니다.
```

Good:
```markdown
설정 항목:
- 데이터소스 URL
- 커넥션 풀 크기
- 타임아웃
- 유휴 연결 수
```

### P6. 코드 블록 전후에 설명 산문

코드 블록만 나열하면 "코드 없이도 글이 읽히는가?" 테스트에 실패한다.

> **근거**: Google Developer Docs — "Don't just show code; explain it."

규칙:
- 코드 블록 **앞**: 이 코드가 무엇을 하는지 (목적)
- 코드 블록 **뒤**: 결과나 주의사항 (선택적)
- 연속 코드 블록 2개 이상이면 사이에 설명 삽입

### P7. 산문 연속 제한

소제목 없이 5문단 이상 이어지지 않는다.

> **근거**: NNG — 소제목은 시각적 앵커 역할을 하며, 스캔 패턴을 안내한다.

### P8. 문단 간 전환

갑작스러운 주제 전환을 금지한다. 논리적 연결어 또는 구조적 전환(소제목)으로 맥락을 유지한다.

판단 테스트: 이전 문단 마지막 문장과 다음 문단 첫 문장을 연속으로 읽었을 때 자연스러운가?

## 2. 헤딩 계층

### H1. 제목은 하나만

H1(`#`)은 문서당 하나. 블로그는 front matter `title`, 위키는 `subject`가 H1 역할.
본문은 H2(`##`)부터 시작한다.

> **근거**: HTML5 문서 아웃라인 모델 — 하나의 `<h1>`이 문서의 최상위 제목.

### H2. 레벨 건너뛰기 금지

H2 → H4처럼 레벨을 건너뛰지 않는다. 반드시 H2 → H3 → H4 순서.

> **근거**: WCAG 1.3.1 — 스크린 리더 사용자가 헤딩 레벨로 문서 구조를 파악한다.
> 건너뛰면 구조적 혼란을 야기한다.

### H3. 빈 헤딩 금지

헤딩 아래에 최소 1문단(또는 하위 헤딩)이 있어야 한다. 헤딩만 있고 내용이 없으면 제거하거나 내용을 추가한다.

### H4. 헤딩 텍스트 형태 통일

같은 레벨의 헤딩은 문법적 형태를 통일한다.

Bad:
```markdown
## 설치 방법
## 설정을 변경하기
## 테스트
```

Good (명사형):
```markdown
## 설치
## 설정
## 테스트
```

Good (동사형):
```markdown
## 설치하기
## 설정하기
## 테스트하기
```

## 3. 목록과 테이블

### L1. 목록 항목 수

3-7개가 적정. 2개 이하는 산문으로, 8개 이상은 그룹화한다.

> **근거**: Miller's Law — 인지 부하 최적 범위 7±2.
> Microsoft Writing Style Guide — "Use a list for three or more items."

### L2. 목록 항목 병렬 구조

모든 항목의 문법적 형태를 통일한다.

Bad:
```markdown
- 설정 파일을 수정합니다
- 테스트 실행
- 배포는 CI/CD를 통해 자동으로 진행됩니다
```

Good:
```markdown
- 설정 파일을 수정한다
- 테스트를 실행한다
- CI/CD를 통해 배포한다
```

### L3. 테이블 용도 제한

테이블은 비교/대조에만 사용한다. 단순 나열에는 목록이 적합하다.

판단 기준:
- 2개 이상 속성으로 항목을 **비교**한다 → 테이블
- 항목을 **나열**만 한다 → 목록
- 항목에 긴 설명이 붙는다 → 목록 또는 정의 목록

### L4. 테이블 열 제한

열 5개 이하를 유지한다. 초과 시 테이블을 분할하거나 목록으로 전환한다.

> **근거**: 모바일 환경에서 5열 초과 테이블은 가로 스크롤이 발생하며,
> 블로그 테마에 따라 레이아웃이 깨진다.

## 4. 코드와 기술 요소

### C1. 코드 블록 언어 명시

모든 코드 블록에 언어를 명시한다. 언어가 없으면 구문 강조가 적용되지 않는다.

```
Bad:  ```
Good: ```java
Good: ```yaml
Good: ```bash
```

### C2. 인라인 코드 용도 제한

인라인 코드(`` ` ``)는 변수명, 파일명, 명령어에만 사용한다. 강조 목적으로 사용하지 않는다.

- 변수/함수: `virtualThreadEnabled`, `handleRequest()`
- 파일: `application.yml`, `build.gradle`
- 명령어: `./gradlew test`
- ~~강조~~: ~~`중요한 내용`~~ → **중요한 내용** (볼드 사용)

### C3. 긴 코드 분할

20줄 이상의 코드 블록은 핵심 부분을 분할하고 각각에 설명을 붙인다.

> **근거**: Google Developer Docs — "Break up long code samples with explanations."
> 독자는 20줄 이상 코드를 한 번에 이해하기 어렵다.

규칙:
- 20줄 미만: 한 블록으로 유지
- 20-40줄: 논리적 단위로 2-3개 블록 분할
- 40줄 초과: 핵심만 발췌하고 전체 코드는 링크/접기 처리

## 5. 시각적 리듬

### V1. 텍스트 밀도 균형

산문 → 목록/코드/테이블 → 산문 순으로 교대 배치한다. 같은 유형이 장시간 연속되면 독자 피로를 야기한다.

> **근거**: NNG — "Visual variety helps maintain reader attention."

Anti-pattern:
```
산문 → 산문 → 산문 → 산문 → 산문 → 코드 → 코드 → 코드
```

Good:
```
산문 → 코드 → 산문 → 테이블 → 산문 → 목록 → 산문
```

### V2. 시각적 변화 주기

스크롤 3회(약 15-20문장) 이내에 시각적 요소(목록, 코드, 테이블, 이미지, 인용)가 등장해야 한다.
산문만 연속되면 이 규칙을 위반한다.

### V3. TL;DR 필수

1500자(약 750단어) 이상의 글은 상단에 TL;DR 또는 요약을 배치한다.

> **근거**: NNG — "Start with the conclusion; busy readers may only read the beginning."

형식:
```markdown
## TL;DR

- 핵심 포인트 1
- 핵심 포인트 2
- 핵심 포인트 3
```

## 6. CJK 특화

### K1. 한영 혼용 공백

영문 단어/코드가 한글 사이에 올 때 앞뒤 공백을 넣는다.

> **근거**: WCAG CJK 타이포그래피 — 한영 경계에 공백이 없으면 단어 경계 인식이 어렵다.
> 다만 인라인 코드(`` ` ``)는 시각적으로 구분되므로 공백 없이도 허용한다.

Bad:
```
Spring Boot에서virtual thread를 활성화하려면
```

Good:
```
Spring Boot에서 virtual thread를 활성화하려면
```

허용:
```
`application.yml`에서 설정합니다
```

### K2. 문장 종결 통일

합쇼체(~합니다/~입니다)로 통일한다. 한다체(~다/~이다)와 혼용하지 않는다.

> **근거**: 톤 일관성은 독자 신뢰의 기본 요소.
> 블로그와 위키 모두 동일 기준을 적용하여 퍼블리시 시 톤 변환 비용을 제거한다.

### K3. 외래어 표기

기술 용어의 첫 등장 시 원어를 병기한다. 이후에는 한글 또는 원어 중 하나로 통일한다.

형식: `한글(English)` — 괄호 안에 원어

예시:
```
가상 스레드(Virtual Thread)는 JDK 21에서 정식 도입되었습니다.
이후 가상 스레드를 사용하면 ...
```

## 7. Blockquote 렌더링

### B1. Blockquote 내 문장 개행

Chirpy prompt blockquote(`.prompt-tip`, `.prompt-info`, `.prompt-warning`, `.prompt-danger`)에서
문장 종결 후 시각적 개행이 필요하면 `<br>`을 사용한다.
마크다운 blockquote는 단일 개행을 무시하므로,
의도된 개행에는 명시적 `<br>`이 필수이다.

> **근거**: 마크다운 blockquote는 `>` 뒤 단일 개행을 연속 텍스트로 렌더링한다.
> Semantic Line Breaks는 소스 가독성 규칙이며 렌더링에 반영되지 않는다.
> 독자가 보는 결과물에서 문장 분리가 의도된 곳에는 명시적 개행 마크업이 필요하다.

규칙:
- 문장 종결(`~입니다`/`~습니다`) 후 아이디어가 전환되면 `<br>` 삽입
- 같은 아이디어의 부연이면 연결 유지 (개행 불필요)
- `<br>`은 종결 문장 끝에 붙이고, 다음 줄은 `>`로 시작

Bad:
```markdown
> **TL;DR** Forward Auth는 인가를 위임하는 패턴입니다.
> 파일 스트리밍 구간에 Auth가 끼지 않아 병목이 없습니다.
{: .prompt-tip }
```

Good:
```markdown
> **TL;DR** Forward Auth는 인가를 위임하는 패턴입니다.<br>
> 파일 스트리밍 구간에 Auth가 끼지 않아 병목이 없습니다.
{: .prompt-tip }
```

## 8. 저자 톤 (Author Tone)

저자의 글이 기계적·구어체·즉흥적으로 읽히지 않게 유지하는 규칙이다.
K2(합쇼체 통일)를 전제로 하며, 그 위에 문장 구조·용어 선택·메타 구성의 일관성을 더한다.

### T1. em dash(—) 금지

한국어 본문에서 em dash(`—`) 사용을 금지한다.
**콜론(`:`)**, **쉼표(`,`)**, **괄호(`()`)**, **줄바꿈** 중 문맥에 맞는 기호로 대체한다.

> **근거**: em dash는 영어권 편집 관행에서 주로 쓰이며, 한국어 문어에서는 읽는 리듬이 끊긴다.
> AI가 생성한 한국어 글에서 과도하게 등장하여 "AI 티"의 대표 신호가 됨.

Bad:
```markdown
Forward Auth는 데이터 경로와 인가 경로의 분리 — 이것이 핵심이다.
```

Good:
```markdown
Forward Auth의 핵심은 데이터 경로와 인가 경로의 분리다.
Forward Auth의 핵심은 데이터 경로와 인가 경로의 분리(decoupling)다.
```

### T2. 어순 전환/두 문장 도치 금지

구어체 목적어 종결(`~다는 걸요`, `~사실을요`)을 금지한다.
또한 "완결 선언 후 목적절만으로 된 뒷 문장"을 덧붙이는 두 문장 도치 구조도 금지한다.
목적어가 필요하면 한 문장 안에서 완결한다.

Bad:
```markdown
나중에야 알게 됐습니다. 이 방식이 정형 패턴이라는 걸요.
나중에야 알게 됐습니다. 이 방식이 정형 패턴이라는 것을.
```

Good:
```markdown
이 방식이 정형 패턴이라는 사실을 나중에야 알게 됐습니다.
```

### T3. 서술어 생략된 체언 종결 금지

"A는 종전과 동일, B는 새 필드." 처럼 서술어가 생략된 명사구 종결 문장을 금지한다.
대비/나열이 필요해도 각 절에 서술어를 붙여 완결한다.

Bad:
```markdown
외부에서 보면 기존 API는 종전과 동일, 신규 API는 새 필드.
```

Good:
```markdown
외부에서 보면 기존 API는 종전 응답을 그대로 유지하고,
신규 API만 새 필드를 노출합니다.
```

### T4. 도입부 메타 설명 중복 금지

도입부에 "이 글이 무엇을 다루는가"를 설명하는 메타 문단을 **1회로 제한**한다.
TL;DR이 이미 메타 요약 역할을 하므로, 본문 진입 전 별도 "이 글은 ~에 중심을 둡니다" 같은 문단을 추가하지 않는다.

> **근거**: 메타 설명이 연속되면 본론 진입이 지연되어 독자가 이탈한다.
> TL;DR + 도입부 한 문단이면 충분하다.

### T5. 즉석 명명 회피, 교과서 용어 우선

정식 용어(업계 표준, 교과서 표기)가 존재하는 개념을 "편의상 X라고 부르겠습니다" 식으로 즉석 명명하지 않는다.
먼저 정식 용어를 탐색하고, 기존 용어가 있으면 그것을 사용한다.
즉석 명명이 불가피한 경우(팀 내 전용 약어 등)에만 명시적 도입을 허용한다.

Bad:
```markdown
이렇게 두 버전이 공존해도 깨지지 않는 상태를 편의상 N-1 호환성이라고 부르겠습니다.
```

Good:
```markdown
이렇게 여러 버전이 공존하는 중간 상태를 **version skew**라고 부릅니다.
이 상태에서 깨지지 않는 성질이 **하위 호환성(backward compatibility)**입니다.
```

### T6. 영어 전문 용어 첫 등장 시 괄호 풀이

K3(외래어 표기) 원칙을 톤 차원에서 강화한다.
`write-both`, `version skew`, `backward compatibility`, `schema-only` 등 영어 전문 용어는
**첫 등장 시 괄호로 한국어 풀이 또는 한국어 표기 + 괄호 원어 병기**한다.
이후 반복은 원어만 써도 무방하다.

Bad:
```markdown
Migrate 단계에서 쓰기는 write-both 기간을 둡니다.
```

Good:
```markdown
Migrate 단계에서 쓰기는 레거시/신규 양쪽에 하는(write-both) 기간을 둡니다.
```

### T7. 기계적 수치 강조체 금지

숫자를 형용사처럼 붙여 강조하는 표현을 금지한다.
`변화 0`, `비용 0`, `위험 0`, `의존 0`, `차이 0` 같은 패턴은 영어식 간결체("zero impact", "no change")를 반사 번역한 결과로,
한국어 기술 문서 관용에 맞지 않아 **공신력을 떨어뜨린다**.

> **근거**: 한국어 기술 문서는 사실 서술을 자연어 문장으로 푸는 것이 관용이다.
> 숫자 0을 형용사처럼 붙이는 표현은 AI 생성 텍스트의 특징으로 인식되어 저자 신뢰도를 손상시킨다.

Bad:
```markdown
외부 계약 변화 0이라 독립 MINOR bump는 불필요합니다.
이 경로는 런타임 의존 0, 빌드 비용 0입니다.
```

Good:
```markdown
외부 계약에 변화가 없어 독립 MINOR bump는 불필요합니다.
이 경로는 런타임 의존이 없고, 빌드 비용도 들지 않습니다.
```

### 유형별 추가 구조 요구사항

#### PoC

PoC 문서는 **결론 선행 + 재현 가능성**이 핵심이다.

필수 구조:
```markdown
## TL;DR (결론: 된다/안 된다 + 핵심 수치)
## 배경 (왜 이 기술을 검증하는가)
## 환경 (재현에 필요한 버전, 설정)
## 검증 (코드 + 실행 결과)
## 결론 (판단 근거, 제약사항, 다음 단계)
```

코드 비중이 높으므로:
- P6(코드 블록 설명)과 C3(긴 코드 분할)이 특히 중요
- 코드와 실행 결과를 분리하여 배치 (코드 → 결과 → 해석)
- P2(문단 길이)는 코드 사이 설명이 1문장이어도 허용

#### 정보성

정보성 문서는 **따라하기 용이성**이 핵심이다.

필수 구조:
```markdown
## TL;DR (선택)
## 개요 (무엇을, 왜)
## 본문 (절차 또는 개념 설명)
## 정리 (핵심 요약 또는 참고 자료)
```

독자가 중간부터 읽을 수 있으므로:
- H2-H4 헤딩이 그 자체로 목차 역할을 해야 함
- 각 섹션이 독립적으로 참조 가능해야 함
- P3(주제문 선행)이 특히 중요 — 원하는 섹션을 빨리 찾게 함

#### 시리즈 어필

시리즈 어필 문서는 **저자 고유 경험 + 스토리텔링**이 핵심이다.

필수 구조:
```markdown
## TL;DR (이 편에서 다루는 것)
## 배경 (이전 편과의 연결 또는 독립 도입)
## 본문 (문제 → 시도 → 결과의 서사)
## 회고 (저자의 판단, 배운 점, 다음 편 예고)
```

포트폴리오용이므로:
- P8(문단 전환)과 V1(밀도 균형)이 특히 중요 — 읽히는 글이어야 함
- Before/After 비교를 적극 활용 (테이블 또는 코드 비교)
- 저자 판단이 드러나는 문장이 각 섹션에 최소 1개
